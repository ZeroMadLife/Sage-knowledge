# V6.6 Runtime 接线与 API 复盘

## Runtime/Engine 接线

- `CodingRuntime` 创建 `TranscriptStore`、`CompactionStore`、`ContextController`。
- Runtime 预写 current user，Engine 使用 `append_user=False`，避免 prompt 与 transcript 重复。
- Engine 的 assistant/tool/error/cancel append 通过 Runtime callback 落 SQLite，并带真实 sequence/message_id。
- `run_finished` 是前端刷新 runs/sessions 的终态事件；断线、取消、pre-engine 异常也必须持久化 terminal run。
- stop 的公开 API 必须携带非空 `run_id`，防旧请求污染新 run。

## API 契约

| 接口 | 目的 | 关键错误 |
| --- | --- | --- |
| `GET /api/v1/coding/{session_id}/context` | 读取 token、effective limit、checkpoint 与 latest attempt | 未配置也返回 200 configured=false |
| `POST /api/v1/coding/{session_id}/context/compact` | 显式手动压缩 | active/busy=409，未配置=422 |
| `GET /api/v1/coding/models` | 返回白名单模型及显式 capability | 不根据模型名字猜窗口 |
| `PATCH /api/v1/coding/{session_id}/model` | 白名单内切换模型并重建 context controller | unknown=422，active=409 |

## 前端状态机

- `context_usage_updated`：更新 used/effective limit/level/compactable。
- `context_compaction_started`：按钮进入 busy。
- `context_compaction_completed`：重取完整 snapshot，不只改 used_tokens。
- `context_compaction_failed`：保留 canonical，reason 映射为中文。
- `run_finished` 不代表 context lease 已释放；前端在 `turn_finished` 后再刷新 snapshot。

## 源码入口

- 后端：`core/coding/runtime.py`、`api/coding.py`、`api/schemas.py`、`api/main.py`
- Engine 事件：`core/coding/engine/events.py`
