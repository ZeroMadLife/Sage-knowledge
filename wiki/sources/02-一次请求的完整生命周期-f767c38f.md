# 一次请求的完整生命周期

> [!abstract] 本章目标
> 以“读取 README 并修正一个错字”为例，从前端点击发送一直追到 diff 展示，不在中途跳到无关模块。

## 0. 前端发送

`CodingComposer.vue::send()` 取得输入内容，调用 Pinia store。Store 确保 session 和 WebSocket 已准备好，然后通过 `CodingStream.send()` 发送：

```json
{"content":"读取 README，修正其中的一个错字"}
```

`CodingStream` 持有一个 `generation`。每次 reconnect/disconnect 都增加代际号，旧 socket 即使晚到消息也会被丢弃，避免切换 session 后旧事件污染新页面。

## 1. WebSocket 接收

入口：`api/coding.py::coding_stream(websocket, session_id)`。

主要动作：

1. 从应用 runtime registry 找到对应 `CodingRuntime`。
2. 接收 JSON 并读取 `content`。
3. 解析可选 slash skill。
4. `async for event in runtime.run_turn(...)`。
5. 每个事件立即 `websocket.send_json(event)`。

WebSocket 只运输事件，不负责决定 Agent 下一步做什么。

## 2. Runtime 建立 run

`CodingRuntime.run_turn()` 首先建立外层生命周期：

```text
检查 active_run_id
  -> 生成 run_id
  -> RunStore.start_run
  -> snapshot_before_run
  -> build_working_memory
  -> 生成 memory_block
  -> turn_started
  -> 创建 Engine
```

这里的顺序很重要。Diff 基线必须发生在模型和工具可能修改文件之前；working memory 必须在 prompt 构建前完成。

## 3. Engine 请求模型

`Engine.run_turn()` 把用户消息写进 history，然后循环：

```text
ContextManager.build
  -> model_requested
  -> model.astream / complete / ainvoke
  -> model_output.parse
  -> model_parsed
```

假设模型返回：

```xml
<tool name="read_file"><path>README.md</path></tool>
```

`model_output.parse()` 将其变成工具 payload。Engine 不直接读文件，而是交给 `ToolExecutor`。

## 4. ToolExecutor 执行读取

执行管线固定为：

```text
normalize payload
  -> 查找 RegisteredTool
  -> Pydantic 参数验证
  -> PermissionChecker
  -> ToolPolicyChecker
  -> 必要时 ApprovalManager
  -> tool_call
  -> RegisteredTool.execute
  -> tool_result
```

`read_file` 的结果被 Engine 追加到 history。下一轮 prompt 因而同时包含用户请求和读取结果。

## 5. 模型提出写入

模型第二轮可能返回：

```xml
<tool name="patch_file">
  <path>README.md</path>
  <old_text>teh</old_text>
  <new_text>the</new_text>
</tool>
```

此时 `PermissionChecker` 根据 permission mode 决定 allow、deny 或 approval required；`ToolPolicyChecker` 检查是否先读后改、是否违反 workspace 策略。通过后工具修改文件，并产生第二组 `tool_call/tool_result`。

## 6. 模型给出最终回答

模型第三轮返回：

```xml
<final>已修正 README.md 中的错字。</final>
```

Engine 产生 `final` 并返回。注意：**final 只表示模型循环结束，不表示 run 的证据已经全部生成。**

## 7. Runtime 生成终态证据

Engine 返回后，Runtime 继续执行：

```text
snapshot_after_run
  -> write diff.json
  -> workspace_diff_ready
  -> run_finished
  -> turn_finished
  -> finally: active_run_id = None + save session
```

`workspace_diff_ready` 只通过 WebSocket 发送摘要。完整 patch 由以下 REST API 按需加载：

```text
GET /api/v1/coding/{session_id}/runs/{run_id}/diff
```

这样大 diff 不会堵塞聊天流。

## 8. 前端归约事件

`CodingStream.onmessage` 把 JSON 交给 `applyCodingEvent(state, event)`：

| 事件 | 页面效果 |
| --- | --- |
| `turn_started` | 显示“开始处理请求” |
| `model_requested` | 显示请求模型、更新 context 字符数 |
| `text_delta` | 增量显示最终文本 |
| `tool_call` | 增加运行中的工具卡片 |
| `approval_required` | 显示审批卡片 |
| `tool_result` | 完成工具卡片 |
| `final` | 完成当前 assistant 消息 |
| `workspace_diff_ready` | 保存 changed files 摘要 |
| `run_finished` | 标记 terminal，刷新 run/session 证据 |

## 为什么要区分三种“结束”

| 事件 | 表示什么 | 能否刷新完整 run 证据 |
| --- | --- | --- |
| `final` | 模型给出了最终答复 | 不能 |
| `run_finished` | diff、trace、状态已经收尾 | 可以 |
| `turn_finished` | turn 流程包装完成 | 通常不作为证据刷新点 |

旧实现若在 `final` 时刷新 run history，就可能在 `workspace_diff_ready` 写入前读到不完整摘要。

## 异常和 Stop 分支

- 模型协议错误：产生 `retry`，最多纠正两次。
- step 超限：产生 `step_limit`，随后仍进入 runtime 收尾。
- 用户 Stop：`request_stop(run_id)` 设置 stop flag，Engine/Executor 在检查点产生 `cancelled`。
- Engine 抛异常：Runtime 转成 `error`，仍产生 `run_finished(status=error)`。
- WebSocket 断开：async generator 被关闭，外层 `finally` 释放 active-run lease。

## 动手

在 `CodingRuntime.run_turn()` 的以下位置临时打断点：生成 `run_id`、收到 Engine event、生成 diff、生成 `run_finished`。完成一次小文件修改，记录真实事件顺序，再撤销断点。
