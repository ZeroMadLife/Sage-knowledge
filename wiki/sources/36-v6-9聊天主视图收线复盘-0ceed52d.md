# V6.9 聊天主视图收线复盘

> 本章记录 V6.9 聊天主视图的实际代码修改与验证证据。它不是新的视觉大重构，也不改变后端的上下文压缩协议。

## 本轮解决的问题

### 1. 长会话不能打断用户阅读

`CodingView.vue` 中的 `message-area` 是唯一的会话滚动容器。此前它只会在接近底部时自动滚动，但用户向上阅读历史后，界面没有明确的新消息状态。

本轮增加两个纯前端状态：

- `isNearBottom`：通过 `scrollHeight - scrollTop - clientHeight < 80` 判断是否跟随；
- `unseenMessageCount`：用户不在底部期间，新进入时间线的消息数量。

当读者在末尾时，新增消息继续跟随；当读者正在上翻时，滚动位置不变，底部显示“`N 条新消息`”和回到底部按钮。点击后才跳到最新位置并清零计数。切换 session 会重置该临时状态，防止会话 A 的未读标记污染会话 B。

```text
新增消息
  |-- 读者在底部 --> 自动滚到底部
  '-- 读者在历史 --> 增加未读数 --> 点击“回到底部” --> 恢复跟随
```

旧的 `loadOlder()` 锚点恢复逻辑未修改，因此向前分页仍会保持当前阅读内容在屏幕中的相对位置。

### 2. Composer 将上下文窗口说清楚

原先 Composer 只显示一个百分比圆环。现在改成预算状态块，使用后端已有的 `CodingContextSnapshot`：

| 后端状态 | UI 状态 |
| --- | --- |
| 未配置模型 | 模型未配置 |
| `context_operation_active` 或本地请求执行中 | 正在压缩 |
| `compactionError` 存在 | 压缩失败 |
| `level=emergency` | 接近上限 |
| `level=compact/high` | 即将压缩 |
| 其余情况 | 正常 |

它显示“已用 token / 有效窗口”、状态标签与进度条，仍保留手动“压缩”入口和 ARIA `progressbar`。本轮没有改动约 65% 开始压缩的服务端门限，也没有新增所谓“无限上下文”。上下文最终仍受模型有效窗口约束，压缩是把可恢复摘要送回上下文，而不是把历史无限塞入模型。

### 3. 过程展示只展示审计阶段

发送消息被 stream 接收后，Store 会立即写入 `thinkingPhase = 准备执行`；随后 server event 将它切换为模型请求、工具、审批或上下文压缩等执行阶段。工具出现时，`CodingThinkingIndicator` 不再被隐藏。

这里展示的是可审计的运行阶段和工具状态，不展示模型原始推理链（CoT）。这是产品安全边界：用户需要知道工作进行到哪一步，不需要也不应依赖内部 token 推理。

### 4. 工具详情可回放且不丢结果

`CodingToolActivity` 现在提供：

- 人类可扫描的工具摘要；
- 默认折叠的完整参数 JSON；
- 结果前 800 字符预览，按需展开完整结果；
- success/warning/danger token 下的图标、spinner 与输出字体。

事件投影层不再把工具结果截断为 2,000 字符。视觉层仍然预览截断，数据层保留完整结果，符合“历史回放和实时事件一致”的目标。

## 关闭的风险

- 用户回看历史时不会被实时消息强制拉到底部；
- 工具运行阶段在发送后立即可见；
- 过程历史不再因前端 2,000 字符截断丢失工具结果；
- 深色主题下工具 spinner、状态图标和过程块使用 Sage token；
- 思考和执行过程块在中心聊天栏正确居中。

## 遗留问题与边界

- 本地浏览器验收只有无后端的空状态：Sage API 未启动时 `/api` 返回 `502`，因此真实运行中回放、断线重连和审批仍需在后端启动后的集成环境执行。
- 生产构建有既有 CSS `:deep` 兼容警告和超过 500 kB 的 chunk 警告；本轮未引入，归入 V6.9.1 的 Markdown 样式与代码分包治理。
- 虚拟长列表、Mermaid、KaTeX、Monaco、终端都不属于本轮；后端的压缩算法与门限也未变。

## 2026-07-13 前端主题浮层收口

Source commit：`7f73e93 fix(sage-v6.9): unify coding overlay themes`（`codex/feat-v6-9-final`）。

用户在深色 Coding Workbench 中打开“权限模式”时，`CodingPermissionModeDrawer.vue` 通过 `Teleport` 挂载到 `body`，但该组件沿用硬编码白底、蓝色选中态和浅灰边框。结果是深色工作台中出现一块不受主题控制的白色浮层。

本次将以下浮层全部收口到 `--sage-*` 令牌：

- 权限模式入口与选择弹层：surface、border、text、hover、selected 和遮罩；
- 计划预览弹层：surface、header、正文、关闭按钮和错误状态；
- 审批卡及其完整 diff 弹层：保持 warning/add/remove 的语义色，同时复用主题 surface、border、shadow 和遮罩。

`frontend/src/style.css` 新增 `--sage-overlay`，浅色和深色各有对应透明度，避免各组件自行写不一致的黑色遮罩。补充 Composer 测试，验证权限弹层可由关闭按钮卸载。

验证证据：

- `npm test -- --run`：`22 files / 235 tests passed`；
- `npm run build`：通过；
- `git diff --check`：通过；
- 生产构建仍报告既有的全局 `:deep()` 兼容警告及超过 500 kB 的 chunk 警告，本次未新增。
- 已于 2026-07-13 fast-forward 合入 `dev/sage-v6`；集成 worktree 中路由聚焦测试 `2 passed`、生产构建和 `git diff --check` 通过。一次并行全量 Vitest 运行中 `src/router/index.test.ts` 的 `beforeEach` 超过 10 秒，单独复跑通过，判定为环境并发超时而非本次行为回归。

关闭风险：深色模式中的权限、计划和审批浮层不再退回白色硬编码样式；后端 API、store、权限契约和 V7 分支均未改动。

遗留边界：自动化浏览器在本机 `127.0.0.1:5175` 只呈现空宿主页，无法获得可信的运行态截图；需在正常桌面浏览器且 Sage API 已启动的集成环境中补做深色模式人工视觉验收。
