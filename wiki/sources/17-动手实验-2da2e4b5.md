# Sage 源码动手实验

> [!abstract] 目标
> 从“看过代码”升级到“能预测行为、能写测试、能定位问题”。每个实验完成后在标题前加 `✅`。

工作目录：

```bash
cd /Users/zeromadlife/Desktop/tour-agent
```

## 实验 0：建立绿色基线

```bash
pytest tests/core/coding tests/api/test_coding_routes.py tests/evals/test_benchmark.py -q
cd frontend
npm run test -- --run
npm run build
```

记录通过数量和耗时。后续实验只跑最小测试，最后再跑基线。

完成标准：知道后端、前端、benchmark 三套验证入口。

## 实验 1：画出真实事件序列

打开：

- `tests/core/coding/test_agent_loop.py::test_agent_loop_user_to_tool_to_final`
- `tests/core/coding/test_runtime_run_lifecycle.py::test_run_turn_emits_run_finished`

先不运行，手写预测事件顺序，再执行：

```bash
pytest tests/core/coding/test_agent_loop.py::test_agent_loop_user_to_tool_to_final -q
pytest tests/core/coding/test_runtime_run_lifecycle.py::test_run_turn_emits_run_finished -q
```

解释为什么 Engine 测试没有 `run_finished`，Runtime 测试却有。

## 实验 2：破坏一次模型协议

在测试里用 ScriptedApiClient 依次返回：

```text
普通散文
普通散文
<final>恢复成功</final>
```

断言 `retry` 数量、`model_requested` 数量，以及协议纠正文本没有写进 session history。

运行：

```bash
pytest tests/core/coding/test_agent_loop.py -q
```

完成标准：能解释 transient protocol correction。

## 实验 3：亲手触发三道安全门

分别构造：

1. plan mode 调 `write_file`，触发 permission deny。
2. 未读文件直接 `patch_file`，触发 policy deny。
3. auto mode 调 `git reset --hard`，触发 approval required。

运行：

```bash
pytest tests/core/coding/test_permissions.py tests/core/coding/test_tool_executor.py -q
```

把三个结果里的 `security_event_type`、`policy_reason` 和 approval event 写进笔记。

## 实验 4：检查 Diff 的证据边界

阅读 `tests/core/coding/test_workspace_diff.py`，然后添加一个测试：

```text
snapshot_before
另一个动作修改同一文件
snapshot_after
```

观察 artifact 能否知道修改者是谁。不要立刻实现归因，先写出当前信息为什么不足。

运行：

```bash
pytest tests/core/coding/test_workspace_diff.py -q
```

完成标准：能区分 change detection 与 attribution。

## 实验 5：跨 Session Memory

跟读：

```bash
pytest tests/evals/test_benchmark.py::test_memory_continuity_recalls_in_second_session -q
```

然后手工检查临时 storage 的：

```text
memory/<workspace_id>/daily/
memory/<workspace_id>/MEMORY.md
memory/<workspace_id>/project-conventions.md
```

尝试另一个 workspace root，确认 ID 和 facts 分离。

## 实验 6：激活 Deferred Tool

让 ScriptedApiClient 先调用：

```xml
<tool>{"name":"tool_search","args":{"query":"memory"}}</tool>
```

第二轮检查 prompt 是否出现 `remember`/`dream` 完整描述，并验证 resume 后 `activated_tools` 仍存在。

运行：

```bash
pytest tests/core/coding/test_engine.py::test_engine_tool_search_activates_deferred_tools_for_next_prompt -q
pytest tests/core/coding/test_todo_plan_worker_runtime.py::test_runtime_persists_activated_deferred_tools -q
```

## 实验 7：理解前端终态竞态

打开 `frontend/src/stores/codingEvents.test.ts`，比较：

```text
final -> terminal false
workspace_diff_ready -> 保存摘要
run_finished -> terminal true
```

临时把 final 改成 terminal true，运行测试观察哪个断言失败，然后撤销修改。

```bash
cd frontend
npm run test -- --run src/stores/codingEvents.test.ts
```

完成标准：能解释刷新过早为什么会漏 diff。

## 实验 8：验证 Worker 的真实能力

阅读：

- `core/coding/multiagent/manager.py`
- `core/coding/multiagent/runtime.py`

回答并用测试证明：

1. `send_message` 是否复用旧 history？
2. `task_stop` 是否传入 Engine `should_stop`？
3. Explore worker 为什么只读？

```bash
pytest tests/core/coding/test_worker_runtime.py tests/core/coding/test_todo_plan_worker_runtime.py -q
```

## 实验 9：运行 Harness Benchmark

```bash
python -m evals.coding.runner
```

打开最新 JSON、Markdown 和 HTML 报告，任选一个 PASS 场景回答：

- 真实走过哪些 Runtime 事件？
- 文件状态由哪个 assertion 验证？
- 如果 Runner 直接调用 Engine，会少什么？
- 这个 PASS 能否证明真实模型也成功？为什么？
