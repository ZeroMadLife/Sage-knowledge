# V6.9 持久时间线与运行重连复盘

## 学习目标

理解“浏览器断开或切换会话后，工具过程为什么还能回来”。这一章的重点不是聊天消息存储，而是把浏览器可见的运行过程变成可排序、可重放的持久事件。

## 本阶段已实现的组件

| 组件 | 责任 | 关键事实 |
| --- | --- | --- |
| `SessionEventJournal` | 记录浏览器可见事件 | 每条事件有稳定 `event_id` 与 session 内单调递增 `sequence` |
| `RunCoordinator` | 持有服务端运行任务 | run 不再依附某一条 WebSocket；浏览器断开不等于取消运行 |
| Timeline REST API | 拉取历史、补齐缺口 | `GET /api/v1/coding/session/{session_id}/timeline?after=&limit=` 支持尾部读取与旧历史分页 |
| Timeline WebSocket | 重连后补发再订阅 | `/stream?after=<sequence>` 先补缺失事件，再进入实时推送 |
| 前端 session 分区 store | 隔离会话状态 | 会话 A 的迟到事件不能写入会话 B |
| `CodingMessageTurn` | 回放统一视图 | user、assistant、model、tool、approval、context、memory、agent、terminal 都可归入时间线 |

## 一条完整调用链

```text
用户发送任务
  -> RunCoordinator 创建并持有 run
  -> Runtime/Tool/审批等产生浏览器可见事件
  -> SessionEventJournal 先写 SQLite（event_id + sequence）
  -> 再经 WebSocket 推给当前浏览器

浏览器切走、刷新或短暂断线
  -> run 继续在服务端执行
  -> 新连接携带 after=最后已见 sequence
  -> 服务端补发缺失 Journal 事件
  -> 前端用 event_id 去重并按 sequence 重建时间线
```

## 关键契约

### 先落盘，再推送

任何可见事件必须先进入 `SessionEventJournal`，再发往 WebSocket。这样推送失败只会让客户端暂时落后，而不会让事件本身丢失。

### 顺序和幂等

- `sequence` 是同一 session 内的排序游标，用于请求“从哪里继续”。
- `event_id` 是稳定身份，用于 REST 回放与 WebSocket 实时事件重叠时去重。
- 旧历史采取有界窗口：实时约 2,000 条，向前可加载的连续窗口最多约 10,000 条，避免无限会话拖垮浏览器。

### 运行所有权

`RunCoordinator` 由服务端而不是 WebSocket 持有运行任务。切换会话只改变前端订阅对象，不得取消旧会话的执行。服务重启后，未完成 run 会以 `interrupted` 形式暴露为可重试状态，不能伪造“自动续跑成功”。

### 前端隔离

前端状态以 `session_id` 为分区键，分别保存 timeline、run 状态、审批、上下文、错误及滚动锚点。处理到达事件前必须核对 `session_id`，防止订阅切换造成串会话。

## 为什么旧方案会让过程消失

旧的历史恢复主要依赖 `role/content`，工具参数、执行结果、审批过程虽然可能曾经持久化，却没有投影回聊天时间线。于是刷新或切回会话时，只能看到最终消息，不能解释模型中间做了什么。V6.9 的 Journal 是“事件事实来源”，`CodingMessageTurn` 是它的可展示投影，两者不可混淆。

## 建议阅读顺序

1. `core/coding/persistence/session_event_journal.py`（或同目录 Journal 实现）了解 schema、sequence 与 cursor。
2. `core/coding/run_coordinator.py` 了解 run 生命周期和取消边界。
3. `api/coding.py` 的 timeline/stream 路由，核对 REST 与 WebSocket 游标契约。
4. `frontend/src/stores/coding.ts` 了解按 session 分区、去重和旧历史合并。
5. `frontend/src/components/coding/CodingMessageTurn.vue` 看事件如何归属到一个对话 turn。
6. 对照 `tests` 中 timeline replay、reconnect、stale terminal 的用例阅读失败保护。

## 已验证证据

- 后端全量测试：`926 passed`（一条既有 LangChain deprecation warning）。
- 前端全量测试：此前集成验证 `195 passed`，生产构建通过。
- 已修复的边界包括：cursor/旧历史分页、重复回放去重、旧 terminal 事件、legacy transcript 兼容和会话滚动锚点。

## 仍需注意的边界

- 这不是“断线后自动恢复模型执行”。服务重启的 run 会标记中断，用户决定是否重试。
- 时间线是单工作区私测能力，不等于多租户事件隔离；租户、队列和持久任务平台属于 V7。
- Journal 保存的是 UI 事件，不替代 canonical transcript、context checkpoint 或 memory canonical store。
