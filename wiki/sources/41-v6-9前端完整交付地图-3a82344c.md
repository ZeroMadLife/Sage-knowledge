# V6.9 前端完整交付地图

> 本章从最终集成点 `dev/sage-v6@2016852` 反查 V6.9 的完整前端交付，不把范围缩成最后一个 Provider commit。用于观察用户可见变化的基线是 `fbbf208..2016852`：前端共 `80 files changed, 6770 insertions, 4531 deletions`。

## 一句话结论

V6.9 不是一次换皮。它把 Sage 从多个旧页面和临时 WebSocket 状态拼成的聊天界面，收束成了一个以 **持久时间线为事实源、会话状态相互隔离、可以恢复运行过程，并由 Provider/模型能力驱动控制面** 的 Coding Workbench。

![[sage-v6.9-frontend-architecture-gpt-image-2.png]]

图由 `gpt-image-2` 生成，用于表达本章的代码结构总结；源码事实仍以本文列出的模块、测试和 source commit 为准。

## 用户现在实际得到什么

### 1. 聚焦聊天的工作台

- `/coding/session/:sessionId` 成为可恢复的会话深链；刷新、从设置页返回和直接打开历史会话不会无条件创建新会话。
- 桌面端是会话栏、聊天主区和临时 Files/Diff/Runs 工具面的工作台；平板和手机改为抽屉，不再长期挤占聊天区域。
- 旧 `ChatView`、`WorkbenchView` 和旧 TourAgent chat store 被移除，主流程统一进入 `CodingView`。
- 视觉 token、浅色/深色 overlay、Markdown、代码高亮和移动端断点统一。

### 2. 一条可读的消息与执行时间线

- 用户与 Sage 使用明确的左右布局和身份标识；optimistic 用户消息先出现，再进入思考、工具和最终回答。
- Markdown 代码块带语言标题、语法高亮和复制按钮。
- 工具调用按单项展示 running/done/error，参数和结果可展开、复制，长结果可截断后继续展开。
- Context、Memory、Agent、Approval 和 Terminal 事件被归入每个 assistant turn 的执行元数据，而不是散落在页面其他区域。
- 用户上翻阅读时停止强制跟随，显示未读数量和“回到底部”；会话切换与历史分页保留滚动锚点。

### 3. Composer 变成真正的运行控制面

- `/skill` 自动补全支持筛选、方向键、Enter 和 Escape。
- 模型切换、权限模式、reasoning 档位、Provider 设置入口、发送/停止集中在底部 command rail。
- reasoning 只展示当前模型 manifest 真正支持的 `off/low/medium/high`，不是前端伪开关。
- 上下文预算显示 used/limit/remaining，并提供服务端手动 compact；未配置模型不会伪造百分比。

### 4. Settings 成为项目配置和观测入口

- Provider 页展示协议、base URL、模型数量、环境变量名和是否已配置，不接收或回显 API Key。
- 本地项目模式可编辑 `.sage/settings.json` 的非敏感描述；部署托管模式只读。
- 模型页展示 provider、上下文窗口和 reasoning 能力。
- 用量页支持 `7d/30d/90d/365d`，展示真实 input/output/cache tokens、请求/会话数、缓存命中率和模型分布。
- Provider 未返回费用时显示 `--`，不会估算或伪造为 0；Provider 与 usage 错误状态互不覆盖。

## 前端架构发生了什么变化

### 界面层

| 入口 | 责任 |
| --- | --- |
| `views/CodingView.vue` | 路由生命周期、会话选择、响应式工作台、滚动跟随与抽屉编排 |
| `chat/CodingMessageTurn.vue` | user/assistant turn、Markdown 和执行元数据容器 |
| `chat/CodingToolActivity.vue` | 工具状态、摘要、参数/结果、复制与长输出 |
| `composer/CodingComposer.vue` | 输入、skill、模型、权限、reasoning、发送/停止 |
| `views/SettingsView.vue` | Provider、模型、用量、Memory、MCP 和外观设置 |
| `chat/CodingContextBudget.vue` | 上下文快照和手动压缩 |

### 状态与投影层

`stores/coding.ts` 不再只维护一组全局聊天字段，而是把消息、时间线、审批、运行状态、上下文和异步 generation 放入 session-isolated state。A 会话的延迟事件或请求结果不能写进当前 B 会话。

`stores/codingTimeline.ts` 把 canonical events 投影成稳定 turn：合并 assistant delta/final，关联 tool call/result/error，恢复 approval 所属关系，并保证每个 run 只有一个 terminal。`codingEvents.ts` 仍承担实时事件的快速反馈；timeline 暂不可用时保留 raw-event message，不把用户眼前的执行过程清空。

### 恢复与并发边界

- 进入已有会话时先加载 timeline，并从 cursor 重放漏掉的页，再连接实时流。
- 恢复 durable run lease 后才 resume，避免页面切换制造第二个 worker。
- resident timeline 和手动 older pages 分别有边界，防止长会话无限占用浏览器内存。
- sessions、usage、runs、memory、model、git、files 等异步请求使用 generation counter，采用 latest-wins。
- pending approval 以 WebSocket 为主、REST polling 为韧性层，并严格写回发起它的 session。

## 最有价值的产品变化

1. **过程可解释。** 用户不只看到最终回答，还能读懂 Sage 正在思考、调用了什么、是否等待审批、工具返回了什么。
2. **历史可恢复。** 切会话、刷新和断线后，timeline 能重建 turn、工具、审批和终态，页面状态不再等同于运行事实。
3. **控制来自真实能力。** 模型、reasoning、context 和 usage 由服务端 manifest/metadata 驱动，前端不再自行猜测。
4. **复杂状态被隔离。** V6.9 最重要的工程价值不是组件数量，而是解决了跨会话串状态、旧请求覆盖新状态和恢复时重复运行。

这使 Coding Runtime 可以继续作为 Sage Learning 的 Practice Engine：学习过程中的实验、工具、错误、审批和产物都有可回放证据，而不是只有一段聊天文本。

## 已关闭风险

- 切换会话时旧事件、旧审批和旧异步响应污染新会话。
- 刷新或断线后只恢复最终文本、丢失工具过程和 run 状态。
- optimistic user message 与 timeline transcript 重复。
- 用户阅读历史时被流式输出持续拉回底部。
- 前端展示模型并不支持的 reasoning，或估算不存在的费用。
- Provider 设置页面泄露、存储或回显 API Key。
- 手机端会话栏、Files 工具面和聊天区互相挤压或横向溢出。

## 遗留边界

- production build 中 `CodingView` chunk 约 `1.07 MB`，仍有超过 500 kB 的分包警告；Markdown/highlight、长时间线虚拟化和页面级拆包属于后续性能治理。
- timeline 手动分页仍以事件页为单位，更自然的按 turn 加载和大历史性能优化尚未完成。
- 本地文件链接与 Sage 文件预览之间还缺统一 deep-link 契约。
- OpenAI/Anthropic reasoning 已有参数契约测试，但当前真实 DeepSeek 目录不支持 reasoning，尚未用相应远端凭据在线 smoke。
- V7 多租户、云密钥托管和部署控制面不属于 V6.9 前端交付，不应倒算为本阶段完成。
