# Session、Run 与持久化

> [!abstract] 本章目标
> 区分 session、run、session event 和 todo 四类状态，知道复盘一次运行应该读哪个文件。

## Session 与 Run

```text
Session：跨多个 turn 的会话状态
Run：一次 user request 对应的执行证据
```

一个 session 可以有很多 runs。Session 负责“接着聊”，Run 负责“这一次具体做了什么”。

## CodingSessionStore

保存：

- workspace_root
- user/assistant/tool history
- runtime mode
- permission mode
- todos
- activated tools
- timestamps

写入采用临时文件 + `os.replace()`，避免进程中断留下半截 JSON。`_safe_session_id()` 拒绝路径分隔符，避免通过 session ID 写到任意位置。

消息回放只返回 user/assistant，tool history 留给 run trace 和 worklog。

## RunStore

每次 run：

```text
start_run
  -> append_trace(event)...
  -> summary from trace
  -> timeline from trace
```

V6 使用 session 分区的 evidence：

```text
<storage_root>/runs/<session_id>/<run_id>/
├── trace.jsonl
├── summary.json
└── diff.json
```

`RunStore` 可以从 trace 派生 status、tool count、changed files 和人类可读 timeline。`run_finished.status` 是终态状态的优先来源。

## Trace 与 Timeline

Trace 保存机器事件，字段完整，适合调试和 benchmark。

Timeline 是 `_timeline_from_events()` 产生的简化展示：

```text
请求模型
读取文件 README.md
工具执行完成
等待审批
运行完成
```

不要只保存 timeline；它丢失了很多原始字段，无法替代 trace。

## SessionEventBus

Session event 文件是粗粒度 JSONL。Runtime 同时：

- 把 run event 写进 run trace
- 把 session 相关事件 emit 到 session event bus
- 把事件 yield 给 WebSocket

它们受众不同：run evidence 用于复盘，session events 用于会话级连续记录，WebSocket 用于实时 UI。

## TodoLedger

Todo 保存在 session state 中，字段包括：

- id
- content
- status
- priority
- note
- timestamps

它不是独立项目管理系统，而是帮助长任务保持执行进度的轻量 ledger。

## `run_finished` 的地位

`final`、`cancelled`、`step_limit` 只是 Engine 的退出原因；Runtime 还需要写 diff、duration 和 tool_steps。

`run_finished` 才表示：

- terminal status 已确定
- duration 已计算
- diff ready 已尝试
- trace 已收尾

前端和报告都应以它为刷新点。

## 当前边界

文件存储适合 V6 单用户私测，优点是透明、易调试；V7 多用户和多实例部署时需要处理并发锁、索引、容量、清理策略和对象存储/数据库选择。
