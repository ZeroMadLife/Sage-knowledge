# Sage 总体架构

> [!abstract] 本章目标
> 能解释 Sage 为什么不是“模型外面套一个聊天框”，并能指出控制面、状态面、证据面分别由哪些组件负责。

## Sage 到底是什么

Sage 是一个 Web 形态的个人 coding agent harness。它把模型、工具、上下文、权限、审批、记忆、运行证据和前端交互装配成一个受控执行环境。

模型只负责提出下一步动作；真正决定动作能否执行、如何记录、何时结束的是 harness。

```text
用户意图
  -> 模型提出动作
  -> Harness 验证并执行动作
  -> 工具结果成为下一轮上下文
  -> 直到最终回答或终止
```

## 三个观察角度

### 控制面：任务怎样往前推进

```text
Vue Coding Workbench
  -> WebSocket API
  -> CodingRuntime
  -> Engine
  -> ToolExecutor
  -> RegisteredTool
```

- `api/coding.py` 接收浏览器请求。
- `CodingRuntime` 建立一次 run 的外层生命周期。
- `Engine` 推进 model -> parse -> tool -> model 循环。
- `ToolExecutor` 在执行前完成参数、权限、策略和审批检查。
- `RegisteredTool` 才真正读取文件、写文件或运行命令。

### 状态面：任务怎样连续工作

```text
Session history
  + runtime/permission/plan mode
  + activated tools
  + todos
  + working memory
  + durable memory
```

- `CodingSessionStore` 保存会话和 history。
- `PlanModeManager`、`TodoLedger` 和 `WorkerManager` 保存运行协作状态。
- `MemoryManager` 把本轮工作记忆与 workspace 级长期记忆注入 prompt。
- `ContextManager` 在预算内组装 system、history、skill、memory 和当前请求。

### 证据面：如何知道 Agent 做过什么

```text
typed events
  -> trace.jsonl
  -> run summary / timeline
  -> diff.json
  -> 前端 Run History / Diff Drawer
```

- `engine/events.py` 定义稳定事件契约。
- `RunStore` 保存每次 run 的 trace，并派生摘要和 timeline。
- `WorkspaceDiffTracker` 对比 run 前后文件快照。
- `run_finished` 是运行证据已经完整的终态信号。

## 顶层模块图

| 模块 | 主要职责 | 不应该负责 |
| --- | --- | --- |
| `api/coding.py` | HTTP/WebSocket 边界、请求解析、响应序列化 | Agent loop |
| `runtime.py` | 对象组装、run 生命周期、trace/diff/terminal event | 解析模型协议 |
| `engine/` | 模型循环、输出解析、step/retry/stop | 直接绕过 executor 跑工具 |
| `context/` | prompt、workspace 安全、compact、diff | 长期事实写入 |
| `tools/` | 工具声明、参数 schema、具体动作 | 决定全局权限策略 |
| `tool_executor/` | 参数验证、permission、policy、approval | 会话持久化 |
| `memory/` | working/durable memory 和 proposal | 代码语义检索 |
| `persistence/` | session、run、event、todo 的持久化 | 调模型 |
| `multiagent/` | worker 生命周期和消息 | 主 run 的前端协议 |
| 前端 store | 连接、事件归约、页面状态 | 推断后端未发送的终态 |

## 一次 run 的核心状态机

```text
idle
  -> active_run_id = run_xxx
  -> snapshot_before_run
  -> turn_started
  -> engine events...
  -> workspace_diff_ready
  -> run_finished
  -> turn_finished
  -> active_run_id = None
```

V6 最关键的工程升级不是多了几个功能，而是终态变得明确：

- 同一个 session 同时只能有一个 active run。
- Stop 可以携带 `run_id`，避免旧请求误停新 run。
- 异常也要产生 `run_finished(status=error)`。
- WebSocket 被关闭时，`finally` 仍要释放 run lease。
- 前端等到 `run_finished` 才刷新 run history 和 diff 证据。

## Web 形态带来的边界

浏览器不能直接扫描用户电脑上的任意目录，也不能直接读取本地 Git 状态。当前 Sage 的 Git 和文件能力来自**服务端配置的 workspace checkout**。

```text
V6：一个服务端 workspace，单用户受控私测
V7：每个受邀用户一个隔离的云 workspace + Git checkout
V8：Local Companion 连接用户本地未 push 的工作区
```

因此“网页里看到 Git 状态”不等于“网页直接读了本地电脑”。当前 `CodingRuntime.git_status()` 是服务器在 `workspace_root` 下运行 Git 命令。

## 设计对标

| 来源 | Sage 吸收的重点 |
| --- | --- |
| Hermes | Runtime 边界、工作台体验、可扩展工具体系 |
| Claude Code / Claw Code | Context 分界、安全治理、计划模式、可测试 harness |
| Pico | 小而清晰的 coding loop 与持久化思路 |
| hermes-studio | Web Coding Workbench 的布局与交互参考 |

对标不是复制代码。Sage 当前采用 FastAPI + Vue/Pinia + typed events，并保留旅游能力作为 domain skill 和 benchmark 场景。

## 第一入口

按顺序打开：

1. `core/coding/runtime.py::CodingRuntime.__init__`
2. `core/coding/runtime.py::CodingRuntime.run_turn`
3. `core/coding/engine/engine.py::Engine.run_turn`
4. `core/coding/tool_executor/executor.py::ToolExecutor.execute`
5. `frontend/src/stores/codingEvents.ts::applyCodingEvent`

## 当前边界

> [!warning] 不要把“测试通过”理解成“已经能公开多用户上线”
> V6 仍是单 workspace、单用户受控私测口径。用户隔离、容器沙箱、配额、认证强化和 Local Companion 属于 V7/V8。
