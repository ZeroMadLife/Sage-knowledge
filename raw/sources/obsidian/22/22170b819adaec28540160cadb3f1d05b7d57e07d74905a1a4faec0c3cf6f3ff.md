# 长短记忆与 Dream 对标研究

> [!abstract] 结论
> Context summary、working memory、durable memory、Code RAG 和技能学习是五个不同系统。Dream 可以委派只读反思子 Agent 生成候选，但不能把对话摘要直接当长期事实，也不能默认静默修改 memory 或 skill。

## 先分清五层

| 层 | 生命周期 | 内容 | 是否自动写 |
| --- | --- | --- | --- |
| Transcript archive | 永久/按保留策略 | 原始用户、模型、工具事件 | 是，事件事实 |
| Working memory | 当前 run/session | 当前目标、todo、最近文件、错误、测试状态 | 是，可从证据重建 |
| Context summary | 一次或多次 compact | 继续当前任务所需的 handoff | 是，但不是长期事实 |
| Durable memory | 跨 session/workspace | 用户确认的偏好、项目约定、重要决策 | 显式 remember 或审批后 |
| Code RAG / AST graph | 随代码变化重建 | 代码 chunk、符号、依赖边 | 索引器维护，不归 Dream |

Skill learning 又是第六种资产：它保存“如何完成一类任务”，需要独立的版本、扫描、审批和回滚。

## Claude Mini Memory

Claude Mini 把 memory 按 cwd hash 隔离，使用四类文件：user、feedback、project、reference。`MEMORY.md` 只是索引，启动时最多注入 200 行/25KB。

每轮 substantial query 会异步执行 side query：

- 至少 2 个中文字符，或 query 中含空白。
- 从最多 200 个最新 memory header 里选最多 5 个。
- 单文件名义上限 4KB，单 session 召回累计上限 60KB。
- 已展示路径不重复注入。
- 超过 1 天的内容带 freshness warning。
- Recall 失败 fail-open，不阻塞主任务。

它最值得借鉴的是“索引不是正文”和“异步、预算化、去重召回”。主要风险是 cwd 移动导致失忆、user preference 被错误地限制在项目内，以及中文字符与 byte/char 门限不一致。

## Hermes 的 MemoryProvider 与后台反思

Hermes 把外部记忆抽象成生命周期接口：

```text
initialize
  -> prefetch / queue_prefetch
  -> sync_turn
  -> on_pre_compress
  -> on_delegation
  -> on_session_switch / on_session_end
```

动态 recall 只注入当前 user turn，不写回历史；session 内 system prompt 使用启动时冻结的 memory snapshot，以保护 prompt cache。

Hermes 的 background review 在主回复完成后运行：默认每 10 个 user turn 检查 memory，累计 10 个工具迭代后检查 skill。同模型可复用 warm cache；不同辅助模型只拿最近 24 条和旧历史 digest。Review fork 只开放 memory/skill 管理工具，禁用压缩、持久 session 和危险命令。

这个机制适合 Sage 的 Dream 委派，但 Hermes 的 skill prompt 倾向“多数 session 至少更新一次”，并允许直接修改 agent-created skill，容易过度学习。Sage 不应照搬直接写权限。

## OpenClaw 的三层 Memory 与 Dream

OpenClaw 的层次更接近 Sage 需要的治理模型：

```text
memory/YYYY-MM-DD.md  -> 工作日记和候选证据
MEMORY.md             -> 精炼、短小、启动加载的长期层
DREAMS.md             -> 人类审阅的 promotion 记录
```

详细日记进入 hybrid vector + keyword 索引，不会每轮整包注入。Dreaming 默认关闭，分 light、REM、deep；只有 deep 能写 `MEMORY.md`，还必须同时满足 score、有效 recall 次数和 query diversity 门限。

Skill Workshop 更保守：`/learn` 只生成 pending proposal，经过 hash binding、安全扫描、审批后才能 apply，并保留 rollback。这个 proposal-first 模型比后台直接写更适合 Sage。

## Sage 当前缺口

当前 Sage 已有 working/durable/dream 的文件骨架，但仍是早期实现：

- Working memory 的 recent file hash 为空，不能判断 freshness。
- Durable selection 只是从 `MEMORY.md` 开头裁 2000 字符，没有 query relevance。
- `workspace_id` 只来自 canonical path；移动目录或不同 worktree 会失忆。
- Dream 只是复制已有 facts 并改成 `proposed`，没有提炼、去重、冲突检测和质量分。
- Proposal 没有完整的 WebSocket、review UI、审计与 rollback 闭环。
- Worker 没有可靠取消、连续 history 和 parent-child trace，不适合直接承担可写自进化。

## 候选治理门限

以下是进入正式设计评审的候选值，不代表已经实现：

| 项目 | 候选门限 |
| --- | --- |
| Dream review cadence | 每 10 个成功完成的 user turn，或用户手动 `/dream` |
| 自动排除 | failed、cancelled、policy denied、无新证据的 run |
| Promotion 最低证据 | 至少 3 次有效 recall、至少 2 种不同 query/context |
| 单次 proposal | 最多 5 条，逐条带 source refs、confidence、reason |
| 自动生效 | V6/V7 默认 0 条；全部 proposal-only |
| Review 子 Agent | read-only transcript/evidence，只能调用 proposal tool |
| Skill proposal | hash binding + scan + approve + rollback；永不直接覆盖 |
| Curator | 7 天检查、空闲 2 小时、30 天 stale、90 天 archive、保留 5 份快照 |

## Dream 正确的数据流

```text
成功 run / explicit /dream
  -> 收集原始 transcript、trace、diff、test、memory provenance
  -> 只读 Reflection Agent 分类：忽略 / 更新 / 新增 / 冲突
  -> MemoryPolicyEngine 计算证据和门限
  -> 生成有版本、有 hash 的 proposal
  -> memory_proposal_ready
  -> 用户逐条 approve / reject / edit
  -> 原子写入 durable store + audit + rollback snapshot
```

Reflection Agent 只能提出“可能值得记住什么”；PolicyEngine 决定“是否允许进入候选”；用户决定“是否长期生效”。
