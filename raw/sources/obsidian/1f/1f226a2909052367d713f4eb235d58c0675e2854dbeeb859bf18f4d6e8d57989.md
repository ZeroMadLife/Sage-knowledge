# 前端 Coding Workbench

> [!abstract] 本章目标
> 看懂 view、store、stream、event reducer 和组件之间的分层，不把所有 WebSocket 逻辑塞回 Vue 页面。

## 四层结构

```text
CodingView.vue              页面编排
  -> useCodingStore         业务状态和异步动作
  -> CodingStream           WebSocket 生命周期
  -> applyCodingEvent       纯事件归约
  -> coding components      具体展示
```

### CodingView

负责三栏布局、消息滚动、打开 plan preview/diff drawer、连接组件事件。它不应该逐个解析后端 event。

### Pinia Store

`frontend/src/stores/coding.ts` 管：

- session 与历史消息
- WebSocket 连接
- thinking/error/context 状态
- permission/runtime mode
- files/git
- skills/MCP
- runs/run detail
- diff artifact

Store 是 orchestration 层：调用 REST、创建 stream，并把事件交给 reducer。

### CodingStream

`CodingStream` 只负责 connect、send、disconnect。`generation` 防止旧 socket 的 onmessage/onclose 污染新连接。

### applyCodingEvent

`applyCodingEvent(state, event) -> effect` 是纯归约边界。它更新传入 refs，并返回：

```text
approvalRequired
toolResult
terminal
```

Store 根据 effect 决定是否再请求 pending approval、刷新文件/Git、刷新 runs/session。

## 为什么 `run_finished` 才 terminal

`final`、`step_limit` 和 `cancelled` 负责完成聊天消息，但 reducer 明确返回非 terminal。

只有 `run_finished` 才返回 `{ terminal: true }`，因为此时后端已经有机会生成 diff 和 run summary。这个细节修复了“run history 刷新过早”的竞态。

`error` 当前也返回 terminal，用于恢复 UI；如果 error 后仍有 run_finished，Store 需要保持幂等。

## Worklog 的数据来源

聊天里的 execution log 来自实时 events；左侧 Runs 的 timeline 来自后端 `RunStore` 派生结果。

```text
实时反馈：applyCodingEvent -> messages.activities
历史复盘：GET run detail -> selectedRun.timeline
```

二者解决不同问题，不能只保存前端临时 activity。

## Diff Drawer 当前实现

当前 drawer：

- 列出 added/modified/deleted
- 标记 binary/sensitive
- 展开显示 unified diff
- 通过 `<pre>` 展示文本

V6 设计书曾规划 lazy Monaco DiffEditor，但当前源码仍是 `<pre>`，所以 Monaco 应标为未完成增强，而不是已实现能力。

## Memory UI 当前实现

Sidebar 有 `/remember`、`/dream` 的入口提示，但没有完整 MEMORY.md 浏览、proposal diff、approve/reject review surface。因此 Memory 后端能力领先于前端产品闭环。

## Responsive

`max-width: 900px` 时左右 pane 被隐藏，只保留中间聊天主区。这保证基本可用，但文件、runs 和 memory 在移动端缺少等价抽屉入口，需要作为产品验收项。

## 文案状态

页面已混合中文与英文，如 Sessions、Skills、Runs、MCP 与中文提示并存。若目标用户以中文为主，应单独做一次可见文案清单，而不是只翻译聊天状态。

## 动手

新增一个只影响 UI 的假事件测试：

1. 先在 `types/api.ts` 加 union。
2. 在 reducer test 构造 event。
3. 实现 reducer 分支。
4. 挂到一个最小组件。

通过这个练习理解事件契约的前端闭环，不必改后端。
