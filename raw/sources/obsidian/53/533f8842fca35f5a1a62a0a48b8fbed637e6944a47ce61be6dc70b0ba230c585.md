## 阶段目标

这一阶段把“用户自己的模型 Provider”和“运行过程可解释性”接入 V7 控制面：用户可以保存经过加密的模型 Provider，Coding Runtime 按账户模型执行，同时在聊天界面展示有边界的运行审计信息与思考状态。

它不是通用的自我进化知识库，也没有实现云端 GitHub 工作区或沙箱；这些仍属于后续阶段。

## 已有实现范围

- 账户级 Provider 与模型配置，支持 OpenAI Chat Completions、OpenAI Responses 与 Anthropic Messages 兼容模式。
- API key 只用独立 `MODEL_PROVIDER_ENCRYPTION_SECRET` 做加密持久化；列表、日志和响应只显示脱敏 hint。
- Provider、模型和默认模型均按 owner scope 查询，不能跨用户读取或修改。
- Runtime 将 `account:{provider_id}:{model_id}` 解析成受控模型实例，不把密钥写进 session、timeline 或 prompt。
- Provider 网络检查具备固定 host、有限超时和错误脱敏边界。
- Run audit 只保留受限摘要，聊天 UI 将 plan、tool、reasoning 和结束状态整合为可阅读的运行轨迹。
- 前端增加 Provider 设置与思考角色动画资源；不代表后端已经暴露完整模型原始思维链。

## 本次测试修复

最初无法收口的原因不是运行时功能失败，而是两处测试基础设施问题：

1. `tests/core/cloud/auth/test_repository.py` 与 `tests/core/cloud/model_providers/test_repository.py` 被 pytest 作为同名模块加载，引发 import mismatch。后者已更名为 `test_model_provider_repository.py`。
2. 生产配置测试遗漏新引入的 `MODEL_PROVIDER_ENCRYPTION_SECRET`，无法证明生产 fail-closed 契约。测试现已显式配置该用途独立的密钥。

## 本地 Conda 运行时依赖修复

PyCharm 使用的 `tour-agent-phase1` 环境安装旧版 `requirements.txt` 后，`api.main` 无法导入：`UsageMetadata` 不再从 `langchain_core.messages` 顶层导出，并且未声明的 `langchain-anthropic` 版本不支持 `ChatAnthropic.thinking` 字段。

`04443ab` 将 LangChain 0.3 组合固定为 `langchain-core==0.3.53` 与 `langchain-anthropic==0.3.12`，保持现有 `langchain==0.3.14`、`langchain-openai==0.2.14` 和 `langchain-community==0.3.14` 的约束。Responses API 适配器改从稳定模块路径导入 `UsageMetadata`。

全量测试同时暴露了一个独立的读取边界问题：SQLite transcript 中的超深 JSON 会泄漏 Python `RecursionError`。现在将其转换为带路径和 message id 的 `TranscriptCorruptionError`，维持损坏证据的统一错误契约。

验证：`tour-agent-phase1` 中 `pip check` 通过，真实 Uvicorn 启动后 `/openapi.json` 返回 54 个接口，后端全量 `1043 passed`，`ruff`、`mypy` 和 `git diff --check` 通过。
