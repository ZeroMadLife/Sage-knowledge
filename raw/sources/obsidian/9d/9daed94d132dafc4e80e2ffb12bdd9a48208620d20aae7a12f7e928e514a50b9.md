## 为什么不应该让用户逐条审核

旧版本把“可追溯”近似实现成了“所有内容都要点击批准”，导致 56 条由本地 Markdown 确定性解析产生的低风险记录长期堆积。可审核不等于每次都要人工审批，正确边界是：

1. 本地私有来源、source revision 未变化、受信 parser、目标路径受限时自动应用。
2. 来源删除或 revision 过期时关闭旧提案，不把旧内容写入 Wiki。
3. OCR、外部模型、回滚、低置信或敏感内容才进入人工确认。
4. 每次自动应用仍保留 ParseArtifact、PolicyDecision、Git revision 和撤销入口。

## 核心实现

`KnowledgeStore.plan_pending_migration()` 是只读风险重评入口。它逐条加载当前来源，比较 source revision，检查解析证据或重新调用本地 parser，最终给出四种 disposition：

- `auto_apply`：重新解析并按 Policy v1 自动提交。
- `retire`：来源缺失或已被新 revision 取代。
- `review`：需要 OCR、外部 parser 或业务判断。
- `block`：敏感内容、证据损坏或目标越界。

计划按全部条目生成稳定 `plan_id`。执行接口必须携带 `expected_plan_id`，服务端在写入前重新计算；任何文件变化都会返回 `409 conflict`。每条执行独立隔离，旧 proposal 使用 revision CAS 关闭，新 proposal 使用现有 ingest、policy 和 Git projection 链路，不存在绕过审计的批量 SQL 批准。

## 已关闭风险

- 旧 proposal 没有 ParseArtifact 时，不再允许直接批量批准。
- 带旧解析证据的 proposal 也必须重新校验当前源 revision。
- 解析器缺失或冲突不会让整个迁移预览返回 500。
- 预览后源文件变化会使 plan_id 变化，旧计划无法执行。
- 单条失败不阻断其他条目，并返回结构化错误码。
- 浏览器响应不暴露 source root 的服务器绝对路径。

## 遗留问题与下一阶段

V7.3.0 解决的是历史审核债务，还没有提供 RAG 检索。V7.3.1 的边界是 revision-aware chunk 与 Hybrid Retrieval：

1. 为每个已批准 page revision 建立稳定 chunk。
2. 本地使用 SQLite FTS，建立可替换 dense retriever 契约。
3. 通过 RRF 融合 sparse 与 dense 结果。
4. citation 必须绑定 source/page revision、block ID 和 chunk ID。
5. 建立增量索引、删除、重建和确定性 benchmark。

Neo4j、GraphRAG、Louvain、HR 公开门面、飞书 Bot、Kubernetes 和公网部署仍不属于 V7.3.1。
