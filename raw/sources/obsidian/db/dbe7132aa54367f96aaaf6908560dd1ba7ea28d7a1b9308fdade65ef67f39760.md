## 解决的问题

此前 Sage 已有 Git Wiki、revision/citation、SQLite 混合检索和可回滚学习，但“知识之间如何连接”没有正式运行时。前端若直接画图只能使用假数据，也无法回答关系来自哪版文档。

V7.5.0 增加独立 Graph Projection：

```text
Git Wiki 当前 page revision
  -> 计算 wiki_watermark
  -> 确定性投影 Page / Source / Concept 等节点
  -> 生成带 citation 的 WIKILINK / EVIDENCED_BY / SHARES_SOURCE
  -> 保存不可变 graph_revision
  -> Graph API 有界读取
```

SQLite 图谱只是派生层。删除图谱表后可以从 canonical Wiki 和当前索引重新生成，不反向修改事实源。

## 快照与水位

每张图由以下输入确定：

- `workspace_id`；
- 当前所有 `page_id + current_revision` 形成的 `wiki_watermark`；
- `projector_id/projector_version`；
- `config_hash`。

同一组输入得到同一 `graph_revision`，重复读取不会产生无意义的新版本。页面更新后产生新 revision，旧快照继续保留，可用于审计和回放。

失败重建先在 SQLite savepoint 内执行。异常时回滚节点、边和证据，只把快照标记为 `error`，不会留下半张图，也不会把内部异常细节写入公开错误字段。

## 节点与关系

当前节点类型：

- `page`：普通 Wiki 页面；
- `source`：原始 Obsidian/Markdown/GitHub 等来源；
- `project/concept/decision/tool`：根据受控 Wiki 目录分类；
- 未解析 `[[wikilink]]`：创建 `missing=true` 的 Concept 节点，作为本地知识缺口。

当前关系：

- `EVIDENCED_BY`：页面指向来源；
- `WIKILINK`：页面中显式 Wiki 链接；
- `SHARES_SOURCE`：两个页面共享同一来源。

任何正式边都至少有一个 `KnowledgeGraphEvidence`，包含：

- `citation_id`；
- `chunk_id`；
- `page_id/page_revision`；
- `source_id/source_revision`；
- 边自身的 `extractor_id/extractor_version/confidence`。

当前页面索引失败、拿不到 chunk 时，投影记录 warning 并跳过关系，不伪造无证据边。

## Graph API

- `GET /api/v1/knowledge/graph/status`
- `POST /api/v1/knowledge/graph/rebuild`
- `GET /api/v1/knowledge/graph`
- `GET /api/v1/knowledge/graph/nodes/{node_id}`
- `GET /api/v1/knowledge/graph/nodes/{node_id}/neighbors`

overview 支持 `graph_revision/q/kind/offset/limit/edge_limit`。响应带 `ETag`，相同 revision 和查询条件使用 `If-None-Match` 可返回 `304`。节点分页后只返回当前页节点之间的边，避免悬空端点；邻域接口最多返回受控数量的一跳关系。

## clean-room 决策

研究了 `nashsu/llm_wiki` 的产品行为和图谱入口，但其仓库使用 GPLv3。Sage 只借鉴“Wiki 链接成图、社区、节点详情、图谱洞察”的能力边界，没有复制 Rust/TypeScript 源码、数据结构、样式或资源。

Sage 与 llm_wiki 的关键差异是：图谱在服务端基于 revision/citation 构建，前端只消费可审计快照，适合未来云工作区和多租户隔离。

## 关闭风险

- 前端不再需要构造假图数据；Graph API 已有真实基础契约。
- 图谱重建失败不会破坏 Wiki、检索索引或最后一个成功快照。
- 旧快照不会偷偷引用新版 citation，证据和 revision 一起冻结。
- API 不返回绝对来源路径或原始私有文档内容。
