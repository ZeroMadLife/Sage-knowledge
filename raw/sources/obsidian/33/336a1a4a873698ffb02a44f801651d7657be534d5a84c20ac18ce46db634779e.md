# API、WebSocket 与事件协议

> [!abstract] 本章目标
> 理解为什么实时进度走 WebSocket，而查询证据和审批动作走 REST；能判断新增事件需要修改哪些地方。

## 两类接口

### WebSocket：持续事件流

```text
WS /api/v1/coding/{session_id}/stream
```

客户端发送 user message，服务端连续发送 typed events。适合 model delta、tool activity、审批等待和终态。

### REST：查询与命令

主要接口：

| 类别 | 接口 |
| --- | --- |
| Session | create、list、resume、messages |
| Workspace | files、file、git/status |
| Approval | pending、respond |
| Run | stop、list runs、run detail、run diff |
| Plan | approve、reject |
| Memory | proposal approve、reject |
| Runtime | model、permission-mode |
| Discovery | skills、MCP servers |

完整 diff 和 run detail 通过 REST 拉取，避免把大对象塞进 WebSocket。

## Session Runtime Registry

`request.app.state.coding_sessions` 保存当前进程内的 `session_id -> CodingRuntime`。REST `_require_runtime()` 和 WebSocket 都从这里取 runtime。

Persisted session 可以 resume 回 registry。Resume 时还要验证保存的 workspace 仍位于服务器配置的默认 workspace 下。

当前 registry 是单进程内存结构，不支持多实例服务直接共享。这是 V7 部署架构需要解决的边界。

## WebSocket 重连

连接建立时，服务端会重新发送：

- 当前非 default runtime mode
- 尚未完成的 plan review

这能恢复一部分控制状态，但当前没有 live run event reattach：如果浏览器断开后 run 仍继续，重新连接不能从 event offset 续播该 run。

## Slash Skill 处理

WebSocket 收到 `/review foo.py` 后：

1. `runtime.resolve_slash(content)`
2. 未知 skill 直接发送 error。
3. 已知 skill 先发送 `skill_invoked`。
4. 原始用户文本进入 history。
5. 展开的 skill body 作为 `skill_prompt` 注入本轮。

这样 history 仍能回放用户真实输入，而不会保存一大段内部 skill instructions。

## Typed Event

后端 `core/coding/engine/events.py` 用 Pydantic 定义事件。主要分组：

```text
生命周期：turn_started, run_finished, turn_finished
模型：model_requested, model_parsed, text_delta, retry, final, step_limit
工具：tool_call, tool_result
审批：approval_required, approval_granted
控制：cancelled, error, runtime_mode_changed, plan_ready_for_review
证据：workspace_diff_ready
记忆：memory_proposal_ready
```

`event_to_dict()` 是 typed model 到 JSON-safe dict 的统一出口。

## 稳定契约的要求

新增事件至少要同步检查：

1. `core/coding/engine/events.py`
2. Runtime/Engine 的产生位置
3. `frontend/src/types/api.ts`
4. `frontend/src/stores/codingEvents.ts`
5. Python event/API tests
6. TypeScript reducer/store tests

只改后端事件而不改前端 union，TypeScript 可能无法表达；只加 type 而不处理 reducer，则事件会被静默忽略。

## 终态顺序

正常编辑 run 的尾部应是：

```text
final
workspace_diff_ready
run_finished
turn_finished
```

出现 error 时也必须尽可能产生 `run_finished(status=error)`。前端以 `run_finished` 的 terminal effect 刷新 runs。

## 当前 Memory 事件边界

`MemoryProposalReadyEvent` 已有后端模型，`dream` 工具也会写 session event bus；但它没有像 Engine events 一样稳定地进入当前 WebSocket yield 链，前端类型和 proposal review UI 也不完整。

因此 Memory proposal 是**后端审批原语已存在、实时产品闭环仍部分实现**，不能只因为 events.py 里有 class 就判定完成。

## 错误处理

- 无 runtime：WebSocket 发送 error 后关闭。
- 非法 user message：发送 error，连接继续。
- 未知 slash skill：发送 error，连接继续。
- Runtime 外层异常：WebSocket 发送 `Coding agent error`。
- REST 资源不存在：404。
- workspace/path 参数错误：400。
