## 本阶段解决的问题

过去的知识库页面只描述未来边界，没有真实 KnowledgeWorkspace。V7.2-P2.1 首次建立了可运行的知识写入状态机：

```text
白名单 Markdown / Obsidian
  -> SHA-256 immutable raw snapshot
  -> Wiki Proposal + unified diff
  -> approve / reject
  -> Markdown Wiki + Git commit
  -> rollback proposal
  -> approve rollback as a new revision
```

这使“自我进化”有了可验证含义：Sage 可以持续提出知识更新，但不能静默修改长期知识；用户拥有批准、拒绝和回滚权。

## 核心模块

### `core/knowledge/store.py`

`KnowledgeStore` 是本阶段的深模块，统一封装：

- source root 白名单和相对路径解析；
- UTF-8 Markdown、2 MiB 上限和符号链接拒绝；
- 高置信密钥模式扫描；
- SHA-256 内容寻址的 raw snapshot；
- SQLite proposal、event、page、revision 元数据；
- optimistic revision conflict；
- 跨进程文件锁；
- Git add/commit 投影；
- 手工修改检测；
- rollback proposal 和 revision history。

API、前端和未来飞书 Source Adapter 都不直接执行 Git 或任意文件读写。

### `api/knowledge.py`

提供真实接口：

- 知识空间摘要；
- 单 Markdown ingest；
- proposal 列表、详情和事件；
- approve/reject；
- page revision 列表；
- rollback proposal。

浏览器只提交 `source_root_id + relative_path`。响应不包含 source root 的服务器绝对路径。云端 tenant isolation 完成前，production 模式 fail closed。

### `KnowledgeView.vue`

知识库页面不再是占位文案，当前可以：

- 查看来源、Wiki 页面和待审核数；
- 从配置好的 Obsidian root 输入相对路径；
- 查看 unified diff；
- 批准或拒绝提案；
- 查看 Git revision；
- 从历史 revision 生成回滚提案。

## Git Knowledge Repository

独立仓库遵循 Karpathy LLM Wiki 的核心边界：

- `raw/sources/`：不可变来源；
- `wiki/`：审核通过的 Markdown；
- `schema.md`：Agent 维护规则；
- `index.md`：内容目录；
- `log.md`：append-only 操作记录；
- `purpose.md`：知识空间目标。

本阶段没有把整个 Obsidian Vault 上传到 GitHub。只初始化了安全结构；真实来源必须先通过白名单和密钥扫描，再逐份审核。

## 已关闭风险

- 浏览器提交任意绝对路径或 `../` 路径穿越；
- 符号链接逃逸 source root；
- 未审核提案直接写入 Wiki；
- 陈旧 proposal 覆盖新 revision；
- `git reset` 式破坏历史回滚；
- 多进程同时审批；
- Git 投影把手工未审核修改一起提交；
- 高置信 API Key 或私钥进入 raw snapshot。

## 当前明确边界

- 首阶段只做单文件 Markdown ingest，不做目录 watch 或批量队列；
- 目前页面是来源投影，不调用 LLM 自动综合概念页；
- 尚未实现 sparse/dense retrieval、RRF、citation bundle 或 benchmark；
- 尚未接入飞书真实 API；
- 应用只创建本地 Git commit，不自动 push Knowledge Repository；
- HR public package 仍保持独立发布边界。

### 飞书 Source Adapter

飞书 CLI 或开放 API 读取文档后必须输出统一 `SourceSnapshot`：source id、document revision、content hash、title、captured time 和 Markdown content。它只负责采集，不直接写 Wiki；后续继续进入同一个 proposal、approve、Git revision 和 rollback 状态机。
