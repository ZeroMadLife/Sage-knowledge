# V6.9 时间线 404 修复与 V7 控制面启动

## 这次遇到的真实问题

页面输入框上方出现：`fetch timeline tail failed: 404`。

它不是聊天内容丢失，也不是浏览器缓存或前端视觉问题。根因是 `dev/sage-v6` 已包含前端的 `fetchCodingTimelineTail()`：

```text
GET /api/v1/coding/session/{session_id}/timeline?tail=true
```

但对应后端的 timeline journal、REST route、WebSocket replay 和 `RunCoordinator` 尚在独立集成分支。因此新会话首次拉取尾部事件时稳定得到 HTTP 404。

## 修复原则：协议必须原子合并

不能只加一个空的 `timeline` endpoint。前端严格要求返回：

```json
{
  "items": [],
  "next_cursor": 0,
  "has_more": false,
  "older_cursor": null,
  "latest_cursor": 0,
  "active_run": null
}
```

同时 WebSocket 必须传递完整 envelope：

```text
event_id + session_id + run_id + sequence + kind + status + timestamp + payload
```

否则 REST 可以从 404 变成 200，前端仍会因缺字段拒绝响应，或实时事件和历史重放具有不同结构。

## 本次代码切片

分支 `codex/fix-v6-9-timeline-contract-atomic` 的提交 `f893d9b` 从已验证的 V6.9 集成分支同步了后端原子切片：

1. `SessionEventJournal`：每个 session 的 SQLite durable event journal，`sequence` 支持补页，`event_id` 支持去重。
2. `RunCoordinator`：run 不归属于一个 WebSocket；socket 断开时 server task 继续，重连订阅 `after` cursor。
3. `/timeline`：支持 forward `after`、末尾 `tail` 和更早历史 `before`。
4. 运行租约和 terminal-once：阻止同 session 双 run，重启后的 unfinished run 记为 `interrupted/retryable`，不假装自动续跑。
5. 同步更新旧 WebSocket 测试：实时事件现在读取 envelope 的 `payload.type`，不能再直接读取顶层 `type`。

## V7 开始方式

V7 不把“买 SSH 服务器”理解为给用户 shell。SSH 只用于管理员部署；用户浏览器访问 HTTPS，后端管理 OAuth、授权、container 和 Git credentials。

V7 分四期：

1. **V7.0 控制面**：邀请制 GitHub OAuth、HttpOnly session、用户/项目/工作区 ownership。
2. **V7.1 云工作区**：GitHub repository 授权、只读 clone、workspace provider、隔离 Git facts。
3. **V7.2 受限执行与部署**：Docker runner、配额、审计、Actions、单机私测部署；随后才能考虑终端。
4. **V7.3 HR 公开窗口**：只读的策展资料包和公开 RAG，绝不查询私有 session、workspace、Git、Memory、secret 或内部 index。

完整实施计划：[[2026-07-13-sage-v7-cloud-control-plane]]（项目仓库 `docs/superpowers/plans/2026-07-13-sage-v7-cloud-control-plane.md`）。
