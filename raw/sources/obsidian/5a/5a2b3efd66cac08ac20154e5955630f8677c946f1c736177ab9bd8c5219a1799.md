# V6 源码复盘与技术债

> [!abstract] 本章目标
> 不用“做完了/没做完”二分法评价 V6，而是按已验证闭环、部分能力和后续版本边界判断。

## 当前基线

- 分支：`dev/sage-v6`
- 当前提交：`2df978a`
- V6.1：run correctness
- V6.2：workspace diff evidence
- V6.3：working/durable memory
- V6.4：10 个 deterministic benchmark
- V6.5：针对复盘问题做 harness hardening

V6 的主线已经从“有一个 Agent Loop”推进到：

```text
可控制：permission / policy / approval / plan
可追踪：run trace / timeline / diff
可记忆：working + durable memory
可验证：scripted runtime benchmark
```

## V6.5 已修复的关键问题

| 问题 | 当前处理 |
| --- | --- |
| Diff 扫描可能跟随 symlink | 扫描时跳过 symlink，并有回归测试 |
| async generator 提前关闭后 run lease 残留 | 外层 `finally` 清理 `active_run_id` |
| 延迟 Stop 可能误停新 run | Stop 可绑定 `run_id` 并拒绝 stale ID |
| 大文件变化可能漏掉 | 全文件计算 hash，内容保存保持有界 |
| truncation 元数据含义错误 | 按 changed count 计算 |
| `/remember` 缺 source_ref | 使用当前 active run ID 写 provenance |
| `/dream` 没有 pending proposal | 增加 proposal、approve、reject 原语 |
| Benchmark 绕过 Runtime | Runner 改为消费 `CodingRuntime.run_turn()` |
| Memory 场景只测同 session | 增加跨新 session continuity |
| 前端刷新 run history 过早 | 等 `run_finished` 再 terminal refresh |

这些修复说明源码评审的价值不只是挑错，而是把设计书中的契约变成测试。

## 当前仍属部分实现

### 1. Diff 归因

Before/after snapshot 能发现时间窗内变化，但还不能可靠区分 Agent、用户和其他进程的并发写入。设计书里的 `external_change` 与 `diff_attribution_rate` 尚未形成完整实现。

### 2. Memory 质量

- `reviewed_at` 字段存在但未形成生命周期。
- durable selection 只是预算裁剪，没有 query relevance。
- working recent file hash 没有填充。
- dream proposal 基本复制已有 facts，缺少合并、去重和冲突处理。
- `memory_proposal_ready` 的 WebSocket 与前端 review 闭环不完整。

### 3. 子智能体

- `send_message` 实际创建新 worker，不是继续旧 history。
- `task_stop` 没有真正绑定 Engine cancellation。
- parent-child run、trace、前端任务树未完成。
- write scope 不是 OS/container sandbox。

### 4. 前端产品完成度

- Diff Drawer 仍使用 `<pre>`，不是规划中的 lazy Monaco DiffEditor。
- Memory 只有命令提示，没有完整索引和 proposal 审批面板。
- 移动端隐藏侧栏和文件区，缺少等价入口。
- 页面可见文案中英文混合。

### 5. Runtime 可维护性

`CodingRuntime` 超过 500 行。V7 增加 workspace provisioning、用户隔离和 sandbox 后，应把 composition root、run coordinator、workspace service 分开。

另外，`test_run_turn_releases_lease_on_aclose` 虽然验证 lease 已释放并通过，但 Python 3.13 测试运行仍报告 `Engine.run_turn.aclose was never awaited`。这说明外层 Runtime generator 提前关闭时，内层 Engine async generator 的资源收尾还不够干净；V6.6 应显式持有并关闭 engine stream，再用 warning-as-error 回归测试确认。

## V6、V7、V8 边界

| 版本 | 目标 | 必须完成 | 明确不做 |
| --- | --- | --- | --- |
| V6 | 单用户受控私测 | run 正确性、diff、memory、benchmark、基本 workbench | 任意用户执行宿主机 workspace |
| V7 | 受邀用户云 workspace | auth、GitHub import、每用户 checkout、sandbox、quota、CI/CD、观测 | 读取用户未 push 的本地改动 |
| V8 | 本地/云混合代码智能 | Local Companion、Code RAG、AST graph、索引更新、公共上线强化 | 将 LLM 推断边冒充 AST 事实 |

## 上线前的建议顺序

```text
V6.6 产品闭环
  -> Memory review UI
  -> Diff/Run 完整验收
  -> 中文文案与 mobile access
  -> 私测 demo script

部署设计
  -> CI quality gate
  -> production image / reverse proxy
  -> secret management
  -> health/readiness/logging
  -> gateway login

V7 workspace isolation
  -> user identity
  -> GitHub import
  -> sandbox and quota
```

不要在 V6 单 workspace runtime 上直接开放“任意用户输入任意仓库路径”。这不是小功能缺失，而是执行边界不同。

## 如何继续复盘

每次 GLM 完成一个阶段，按以下顺序审：

1. 设计契约是否真的落在稳定边界。
2. 失败、Stop、断连、并发是否有测试。
3. 前端是否等到正确终态。
4. Artifact 是否能证明行为，而不只是 UI 显示成功。
5. Benchmark 是否穿过真实 Runtime。
6. 文档中的“已实现”是否有源码和测试共同支撑。
