# Sage 个人 AI 学习助手重定位与路线校准

## 本次研讨结论

Sage 的长期主产品不再定义为 Personal Web Coding Agent，而是：

> **Sage — Personal AI Learning Companion**
>
> Sage 是一个面向技术学习与个人成长的 AI 学习助手。它连接用户的代码、Obsidian、文档、课程、论文与面试材料，通过检索、规划、实践、验证、复盘和长期记忆形成个人学习闭环。

Coding 能力继续保留，但从“产品本身”调整为学习系统中的 **Practice Engine**：用于源码阅读、代码实验、项目实践、测试验证和能力证据采集。Sage 不再把“追平 Codex 或 Claude Code 的通用 coding harness”作为差异化目标。

本章是产品与路线研讨结论，不代表 V7/V8 功能已经开发。当前源码事实仍以 `dev/sage-v6`、测试与 [[30-V6.9阶段产出与Git收口记录]] 为准。

## 为什么调整定位

Codex、Claude Code 等产品已经用强模型和成熟 harness 覆盖大部分通用 coding agent 能力。Sage 如果继续把重点放在“多几个 coding tool、UI 更像 Claude Code”，会进入高投入、低差异的正面竞争。

Sage 已经积累的 Context、Memory、Proposal、Transcript、Run Trace、Diff、Skills 和 Benchmark 更适合解决另一个长期问题：

> 用户不缺一次性的答案，缺的是一个能理解个人资料、维持学习目标、安排实践、验证掌握度并持续复盘的系统。

因此，Sage 的竞争边界从“替用户写代码”转为“帮助用户真正掌握知识并留下成长证据”。

## 产品核心闭环

```text
知识摄取
代码仓库 / Obsidian / 文档 / 课程 / 论文 / 面试记录
        ↓
知识建模
概念、技能、依赖关系、来源、掌握度、遗忘风险
        ↓
目标与诊断
学习目标、已有能力、知识缺口与可用时间
        ↓
学习计划
阅读、讲解、代码实验、练习题、项目任务
        ↓
实践与验证
Coding、问答、测试、费曼讲解、真实产物
        ↓
复盘沉淀
错误归因、能力证据、Memory Proposal、下一步调整
        ↺
```

检索到资料不等于学会知识。Sage 后续需要区分以下状态，而不是只记录“聊过这个主题”：

1. 资料中存在；
2. 用户已经接触；
3. 用户能够解释；
4. 用户能够在提示下应用；
5. 用户能够独立应用；
6. 间隔一段时间后仍能回忆和迁移。

## 现有模块的新职责

| 现有模块 | 原 Coding Agent 叙事 | 学习助手中的职责 |
| --- | --- | --- |
| Context | 防止 coding 对话超出窗口 | 保持长期学习任务与当前实践连续性 |
| Durable Memory | 记住项目约定 | 保存用户目标、偏好、稳定误区与审核过的能力事实 |
| Dream Proposal | 整理运行记忆 | 从学习证据中提出可审核的知识状态更新 |
| Skills | `/review`、`/test` 等工作流 | 阅读、费曼讲解、练习、复盘等教学策略 |
| Transcript / Run Trace | 工具执行审计 | 学习过程、尝试、错误和能力判断的证据 |
| Coding Runtime | 代码任务执行器 | Practice Engine：生成实验、运行测试、验证项目实践 |
| Benchmark | Agent 是否完成任务 | 检索质量、学习任务完成度和掌握度判断是否可靠 |
| Knowledge Graph | AST/符号关系 | 连接概念、资料、代码、项目、能力与学习前置依赖 |

## 必须保持分离的系统边界

以下系统不能因为都与“记住知识”有关就塞进 `MemoryManager`：

| 系统 | 回答的问题 | 更新机制 |
| --- | --- | --- |
| Context Summary | 当前任务怎样继续执行 | 对话边界压缩，可丢失、可重建 |
| Durable Memory | 用户明确确认过哪些稳定事实 | 有 provenance、proposal 与 revision |
| Personal Knowledge RAG | 当前问题应该检索哪些个人资料 | 文档索引随知识源增量更新 |
| Code RAG / AST Graph | 代码现在是什么、符号怎样关联 | 随仓库 revision 重建或增量更新 |
| Learning State | 用户对知识点掌握到什么程度 | 只能由可验证学习证据更新 |

学习状态尤其不能由模型一句“用户应该懂了”直接修改。它至少需要关联 quiz、解释、代码实验、测试结果或人工确认之一。

## 第一条产品主流程

第一版不从“万能个人助手”开始，而从最能复用现有资产的流程开始：

> **导入一个代码仓库及相关 Obsidian 笔记，Sage 帮用户理解项目、建立知识地图、制定学习路径、安排代码实验，并持续记录经验证的掌握度。**

建议的用户旅程：

1. 用户选择一个学习主题、代码仓库和 Obsidian 资料范围；
2. Sage 展示来源清单和索引状态，不在后台无边界读取整个知识库；
3. Sage 生成“已有知识 / 前置知识 / 待学习 / 可实践”地图；
4. 用户确认一个阶段目标，例如“理解这个 Agent Runtime 并能独立解释一次 run”；
5. Sage 组合阅读材料、源码调用链、解释问题和最小代码实验；
6. 用户通过回答、修改代码、运行测试或费曼讲解提交学习证据；
7. Reviewer 判断证据是否支持掌握度变化，并产生可审核 proposal；
8. 用户确认后更新学习状态，Sage 安排下一步或间隔复习。

## V6.9、V7、V8 路线校准

### V6.9：可信 Agent 底座，继续完成

V6.9 的持久时间线、运行重连、Context、Memory Proposal、Skills 权限和只读 Agent 仍然是学习助手必需的底座，不因定位调整而推倒重做。

必须继续按既有收口顺序完成：

1. `ToolExecutor.allowed_tools` 强制执行；
2. YAML Skill manifest 与确定性路由；
3. 只读 Explore / Reviewer 事件、取消和结构化结果；
4. Dream 保持 proposal-only，不能获得通用写权限；
5. 修正 README 和旧 Obsidian 中仍以 Coding Agent 为主产品的陈旧叙事。

### V7：Learning MVP，替代“先做云平台化”的优先级

V7 的核心目标调整为验证学习闭环，而不是优先完成多租户 SaaS 基础设施。

建议范围：

- 学习主题 Workspace 与目标管理；
- Obsidian / Markdown / 代码仓库的受控数据源接入；
- 基础 Hybrid Retrieval 与可点击来源引用；
- 学习计划、学习 session、练习和复盘；
- 最小 Learning State 模型与 evidence contract；
- Coding Runtime 作为 Practice Skill；
- 本地单用户 sandbox，用于安全运行代码实验。

原 V7 中的 auth、tenant、quota、云 workspace、CI/CD 和公网多实例部署降为后置项。Terminal 仍必须依赖 sandbox、权限和审计，不能因为学习产品需要代码实验就直接开放宿主机 shell。

### V8：Personal Knowledge Intelligence

V8 承接原有 Code RAG / AST Graph 规划，并扩大到个人知识源：

- Markdown、Obsidian、代码、课程、论文和面试材料增量索引；
- BM25 + dense retrieval + symbol retrieval；
- RRF 融合、rerank 与来源 provenance；
- AST graph 与概念 knowledge graph 分层；
- 跨来源实体消歧和图邻居扩展；
- Local Companion 读取用户明确授权的本地未提交代码；
- Retrieval / Generation / Learning 三层 benchmark；
- 面向公开使用的索引隔离、秘密扫描和安全强化。

第一版不为了简历关键词同时引入 Elasticsearch、Milvus、Neo4j 和 Kafka。优先验证检索问题本身：

```text
SQLite FTS5 或等价 BM25
+ Qdrant / pgvector 二选一
+ AST / Graphify 可重建图
+ RRF
```

只有数据量、更新吞吐或查询模式证明单机方案不足后，才升级基础设施。

### V9 候选：Adaptive Learning

- 掌握度与遗忘曲线；
- 间隔重复和主动回顾；
- 自适应练习难度；
- 错误模式与知识缺口归因；
- 项目制学习、面试模拟和迁移练习；
- 长期学习效果评估。

V9 只是方向占位，不进入当前实现计划。

## 量化指标重构

现有 deterministic benchmark 使用 `ScriptedApiClient`，可以证明 harness 回归稳定，但其中 `task_completion_rate=1.0` 和 `p95=7ms` 不能代表真实模型质量。

学习助手需要三层指标：

### Retrieval

- `Recall@K`、`MRR`、`NDCG@K`；
- source coverage 与 citation correctness；
- 增量索引 freshness；
- 代码 symbol / 文档 chunk / graph neighbor 分路贡献。

### Agent / Harness

- task completion、tool success、policy compliance；
- recovery rate、approval rate、P95 latency；
- context compaction continuity；
- run evidence completeness。

### Learning Outcome

- pre-test / post-test 增益；
- delayed recall；
- independent application rate；
- misconception correction rate；
- plan completion 与复习命中率；
- learning-state calibration：预测掌握度与真实验证结果的偏差。

在没有真实用户数据前，只能称为 benchmark design 或离线实验，不写“学习效率提升 X%”。

## 明确不做

- 不把 Sage 做成日历、邮件、购物、生活管理全覆盖的通用助手；
- 不继续以复刻 Claude Code / Codex 全部 coding 能力为主目标；
- 不把 RAG、Memory、知识图谱和 Learning State 合并成一个巨型模块；
- 不从聊天次数推断用户掌握度；
- 不在 V7 Learning MVP 之前优先投入多租户计费和大规模云调度；
- 不为了技术名词堆叠提前引入 Kafka、Milvus、Elasticsearch、Neo4j 全家桶。

### 任务类型

产品架构研讨与阶段计划，不直接开发。

### 必读材料

### 已确定决策

- Sage 主产品改为 Personal AI Learning Companion；
- 服务对象优先是程序员和技术学习者；
- 数据范围最终覆盖代码、Obsidian、文档、课程、论文和面试材料；
- Coding Runtime 是 Practice Engine，不再是主产品本身；
- V7 优先 Learning MVP，本地单用户安全闭环先于多租户 SaaS；
- V8 建 Personal Knowledge Intelligence，保留 Code RAG、AST Graph 和 Local Companion；
- 当前讨论不代表 V7/V8 已实现。

### Plan Mode 需要产出的内容

请先研讨并形成一个分阶段方案，至少覆盖：

1. 第一条端到端学习主流程和非目标；
2. `LearningWorkspace`、`KnowledgeSource`、`LearningGoal`、`LearningActivity`、`LearningEvidence`、`LearningState` 的最小领域模型；
3. 现有 Context、Memory、Skills、Run Trace、Reviewer、Coding Runtime 如何复用，哪些边界不能复用；
4. V7 Learning MVP 的 3 至 4 个可独立验收里程碑；
5. V8 检索与知识图谱的数据流、索引边界和技术选型门槛；
6. Retrieval / Harness / Learning Outcome 三层 benchmark；
7. 数据隐私、来源授权、prompt injection、秘密扫描和 sandbox 边界；
8. 旧 V7 云平台化任务哪些保留、推迟或删除；
9. README、Obsidian 和面试叙事迁移清单。

### Plan Mode 约束

- 先读当前源码和测试，不根据旧 README 猜实现状态；
- 不修改产品代码；
- 不把设计项写成已交付；
- 不默认引入大型基础设施；每个组件都要给出数据规模或能力门槛；
- 不开始实现，直到用户确认学习主流程、领域模型和 V7 里程碑；
- 输出中文设计与计划，代码标识符和 API 字段保留英文。

### 第一轮需要回答的问题

1. 第一版学习主题是否只支持“代码仓库 + Obsidian”，还是同时支持 PDF / 课程转录？
2. 掌握度更新以哪些 evidence 为最低可信门槛？
3. 学习计划是用户显式创建，还是允许 Sage 主动提出但必须经用户确认？
4. V7 是否继续保持单机本地产品，还是需要保留最小远程访问能力？
