# 多智能体系统

> [!abstract] 本章目标
> 看懂 Sage 当前子智能体是一个轻量 background worker，而不是完整的分布式 agent 编排平台。

## 入口

Deferred tools：

- `agent`
- `send_message`
- `task_stop`

它们通过 `ToolContext.worker_manager` 调用 `WorkerManager`。

## WorkerTask

每个任务保存：

```text
id
description
subagent_type
write_scope
prompt
thread
status
result
timestamps
```

允许的类型：

- `worker`：可以在指定 write scope 内写。
- `Explore`：plan/read-only 模式。

## 启动过程

```text
主 Agent 调 agent tool
  -> WorkerManager.spawn
  -> 创建 daemon thread
  -> asyncio.run(run_worker_task)
  -> 新 model client
  -> 新 ContextManager + Engine
  -> max_steps = 20
  -> result 写回 WorkerTask
  -> notification queue
```

Worker 共享同一个 `WorkspaceContext` 根路径，但拥有独立 Engine、model instance 和 history。

## write_scope

Worker 的 `PermissionChecker` 接收 write scope。文件写入必须位于 scope 下；Explore worker 则直接进入只读 plan mode。

这是当前最重要的子 Agent 隔离，但它不是操作系统沙箱。Worker 仍在同一个 Python 进程、同一服务端文件系统中。

## `send_message` 的实际语义

当前 `WorkerManager.send_message()` 并不是继续原线程里的对话历史，而是读取旧 task 的 description/type/scope，再 `spawn()` 一个新任务处理新 message。

因此它更接近“按同一配置创建后续 worker”，不是具备连续 mailbox/history 的真正 resume。

## `task_stop` 的实际语义

`stop()` 只把状态改为 `stopping`。`run_worker_task()` 的 Engine 当前没有绑定这个 status 的 `should_stop` 回调，因此底层 worker 不一定立即停止。

这是需要明确记录的部分实现，不应在产品文案里声称“可可靠取消子 Agent”。

## 通知

Worker 完成后向线程安全 queue 写入字符串。主 Runtime 可以 drain notifications，但当前事件协议和前端对子 Agent 树、进度、结果的展示仍较弱。

## V6 与后续边界

V6 当前适合：

- 独立只读调查
- 明确目录范围的小任务
- 后台返回一个 final 结果

V7/V8 若要成熟，需要：

- 真正的任务取消信号
- 连续 worker conversation
- run/trace 与 parent-child 关联
- workspace 并发写冲突处理
- 前端任务树和事件
- 资源配额与 sandbox
